# ADR-005: First-Run Signup Screen with Self-Closing Registration

## Status

Accepted

## Date

2026-05-20

## Context

WineCollector is single-user in the MVP, but it ships with `fastapi-users` and is multi-user-ready architecturally. The MVP needs a way to create the first (and only) user without either (a) requiring the user to run a CLI command outside the app, or (b) leaving a public signup endpoint exposed indefinitely.

The three obvious options are: a CLI bootstrap script, a permanent signup page, or a self-closing first-run signup screen.

## Decision

Implement a **first-run signup screen** that is accessible at `GET /signup` only while the `users` table is empty. After the first user is created:

- The route returns HTTP 404 (not redirect) — the signup surface effectively disappears.
- Login at `GET /login` becomes the only public entry point.
- `fastapi-users`' default `/auth/register` route is **not mounted** in the app — registration is exclusively handled by the custom first-run route.

The check "no user exists yet" is a `SELECT 1 FROM users LIMIT 1` performed inside the signup route's handler. If a user already exists, the handler returns 404 immediately.

## Alternatives Considered

### Alternative 1: CLI script (`poetry run create-user`)

- **Description**: A Click/Typer command that prompts for email and password, hashes the password, inserts a user row.
- **Pros**: zero attack surface — the app never exposes a signup endpoint.
- **Cons**: requires the user to drop to a terminal on a freshly-deployed app; doesn't work cleanly on a hosted environment where the user only has UI access.
- **Why rejected**: extra friction at the most fragile moment (first run); the user is a single non-technical-deployment scenario person who values the UI.

### Alternative 2: Permanent signup page

- **Description**: `GET /signup` remains available forever.
- **Pros**: trivial to build.
- **Cons**: contradicts the single-user MVP — a leaked URL or default open registration is the kind of mistake the product should not allow.
- **Why rejected**: exposes an unnecessary attack surface for zero benefit.

### Alternative 3: Seed from environment variables

- **Description**: `INITIAL_USER_EMAIL` and `INITIAL_USER_PASSWORD` in `.env`; app creates the user on first startup.
- **Pros**: fully automated.
- **Cons**: plaintext password in `.env`, even if temporary; requires editing `.env` for the first run; awkward to rotate.
- **Why rejected**: secrets handling smell; password ends up in shell history or backups.

## Consequences

### Positive

- One-click first-run experience through the browser.
- No admin commands or environment variables to manage.
- The signup route disappears the moment it's no longer needed — no permanent public registration.

### Negative

- One extra route + template (`templates/auth/signup.html`) on top of the login flow.
- A racy `SELECT` before insert in a multi-replica setup could theoretically allow two concurrent signups. Acceptable for single-replica MVP.

### Risks

- **Risk**: someone deploys the app, leaves it idle, and an attacker reaches `/signup` before the legitimate user does.
  - **Mitigation**: the app is intended for personal/home deployment. Document in the README that the first action after deploy is to visit `/signup`. Future improvement (Phase 2+): a one-time setup token printed in the logs on first run.

## Implementation Notes

- The first-run check is wrapped in a small dependency: `async def signup_open(db: AsyncSession) -> bool` returns whether the users table is empty.
- The signup handler returns the login page (with a success flash) after creating the user — never the signup page again.
- The password hashing uses fastapi-users' default `PasswordHelper` (Argon2 via `passlib`).
