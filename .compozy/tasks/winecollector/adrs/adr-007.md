# ADR-007: ILIKE-Based Search in MVP (No Full-Text Index)

## Status

Accepted

## Date

2026-05-20

## Context

The PRD requires free-text search across wine `name`, `winery`, and `region`, plus filters for grape, wine type, and stock availability. The data volume is bounded by the use case: a single user's personal cellar — tens to a few hundred wines, with a long-term ceiling probably under 2,000 rows.

PostgreSQL offers three reasonable search strategies for this scale:
1. `ILIKE '%term%'` with optional `pg_trgm` GIN indexes.
2. `tsvector` full-text search with `to_tsquery` and a GIN index.
3. `tsvector` + `pg_trgm` combined for fuzzy matching.

The trade-off is engineering investment (extensions, generated columns, triggers, query construction) vs. user-visible quality at this scale.

## Decision

Use **`ILIKE '%term%'`** across `name`, `winery`, and `region` joined by `OR`. No PostgreSQL extension is required. No GIN index in V1 — sequential scan over a few hundred rows is sub-millisecond.

The query construction:
- The search term is lowercased and surrounded by `%` for substring matching.
- Empty search returns the full list filtered by the other criteria (grape, type, stock, sort).
- Filters are composed as additional `WHERE` clauses (`type = :type`, `:grape = ANY(grapes_array)` once grapes have a structured representation, `stock > 0` when "only available" is on).

## Alternatives Considered

### Alternative 1: tsvector full-text search

- **Description**: Generated `tsvector` column over `name || winery || region`, GIN index, queries via `to_tsquery` with Portuguese stemming.
- **Pros**: stemming (e.g., "vinhos" matches "vinho"), ranking, scales to millions of rows.
- **Cons**: requires a generated column, GIN index, query construction with `plainto_tsquery` for safety, more brittle ergonomically (multi-word terms need careful handling); zero user-visible benefit at this scale.
- **Why rejected**: massive over-engineering for a personal cellar. Substring matching on a wine name is exactly what the user wants ("cas" matches "Casillero del Diablo").

### Alternative 2: `pg_trgm` + GIN for fuzzy matching

- **Description**: Enable `pg_trgm` extension, GIN index on the searchable columns, query with `LIKE` (the extension makes `ILIKE` use the index).
- **Pros**: tolerates typos ("casiyero" matches "casillero").
- **Cons**: an extension to install + maintain; the user typed the name themselves or it came from the scraper — typo tolerance is nice-to-have, not required.
- **Why rejected**: deferred to Phase 3 if real usage shows typos are a problem.

## Consequences

### Positive

- Zero extensions, zero indexes, zero generated columns to manage.
- Search code is two SQLAlchemy expressions and a parameter.
- Easy to read, easy to test.

### Negative

- No stemming — "vinhos" does not match "vinho". Acceptable because users search by proper nouns (wine name, winery).
- Sequential scan if the cellar grows beyond ~10,000 rows. The current ceiling is ~2,000; revisit then.

### Risks

- **Risk**: cellar grows beyond comfortable ILIKE scan range.
  - **Mitigation**: monitoring of cellar size is implicit in the cellar overview page; if scans become slow (>200ms), add a trigram GIN index in a small migration. The change is purely additive.

## Implementation Notes

- The search service constructs `select(Wine).where(or_(Wine.name.ilike(pat), Wine.winery.ilike(pat), Wine.region.ilike(pat)))` with `pat = f"%{term.lower()}%"`.
- Filters are composed via `where()` chained calls; sort via `order_by(desc(Wine.purchased_at))` or `order_by(desc(Wine.vintage))`.
- Pagination is not implemented in V1 — the typical cellar fits on a single page. If the cellar grows past 200 rows, add cursor pagination in a future iteration.
