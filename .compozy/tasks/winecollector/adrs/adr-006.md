# ADR-006: WineTasting Schema Created in V1, UI Deferred to Phase 2

## Status

Accepted

## Date

2026-05-20

## Context

The PRD (per ADR-001) defers the tasting-notebook UX to Phase 2. However, AGENTS.md and ADR-001's implementation notes both call out a desire to "anticipate `WineTasting` 1:N from V1" to avoid a painful migration when Phase 2 starts. The technical question is whether the V1 codebase carries the model and migration for `wine_tastings` at all, given there is no UI consuming it.

There are two tensions: strict YAGNI says "don't write what you don't use", but a deferred migration that has to be designed under time pressure during Phase 2 is exactly the kind of change that breaks data later.

## Decision

In V1 we ship the **`wine_tastings` table, its Alembic migration, the SQLAlchemy model, and the Pydantic schemas** for tasting records. No router, no service, no template references the model. The frontend never reads or writes it. Phase 2 adds only the router, service, and templates — the schema is already in place.

The model:
- `WineTasting.id` (PK, integer)
- `WineTasting.wine_id` (FK to `wines.id`, indexed, `ON DELETE CASCADE`)
- `WineTasting.tasted_at` (date, not null, default today)
- `WineTasting.notes_visual` (text, nullable)
- `WineTasting.notes_olfactory` (text, nullable)
- `WineTasting.notes_palate` (text, nullable)
- `WineTasting.memories` (text, nullable)
- `WineTasting.created_at` / `updated_at` (timestamps)

The `Wine` model gets a `tastings: Mapped[list[WineTasting]]` relationship with `lazy="selectin"` from the start.

## Alternatives Considered

### Alternative 1: Pure YAGNI — model lives in Phase 2

- **Description**: V1 has only `users` and `wines`. Phase 2 introduces the migration, model, and UI together.
- **Pros**: smaller V1 codebase.
- **Cons**: a migration designed alongside the UI tends to leak UI assumptions into the schema; reviewing a schema in isolation before Phase 2 starts catches design smells earlier; running a fresh migration on production-like data adds risk later.
- **Why rejected**: the marginal cost of one extra migration file + model + schemas in V1 is tiny; the benefit of having a stable schema reviewed and tested before Phase 2 begins is real.

### Alternative 2: Single denormalized `tastings` JSONB column on `wines`

- **Description**: Store all tastings as a JSONB array inside the `wines` row.
- **Pros**: zero extra table.
- **Cons**: loses relational queries (e.g., "all tastings in the last 30 days"); pagination of tastings on a single wine becomes JSON manipulation; UI in Phase 2 has to deserialize and re-serialize on every edit; doesn't extend cleanly to per-tasting metadata (photos, ratings).
- **Why rejected**: Phase 3 wants stats like "how many tastings logged this month" and "average frequency of opening". A relational table makes those queries trivial.

## Consequences

### Positive

- Phase 2 begins with a stable schema already in production.
- The export feature (V1) naturally extends to include tastings in Phase 2 (`schema_version` bumps from 1 to 2).
- The data model is reviewed in calm conditions during V1, not while implementing a UI in Phase 2.

### Negative

- The V1 codebase carries unused model/schema files — minor code surface for an inactive feature.
- Integration tests need to exercise the model only via direct SQLAlchemy calls (no HTTP path).

### Risks

- **Risk**: the schema designed in V1 turns out wrong when Phase 2 starts.
  - **Mitigation**: keep the schema minimal — only fields that are clearly needed for "1 vinho → N degustações com notas + memórias". Anything else is a Phase 2 migration anyway.

## Implementation Notes

- The model file lives at `src/winecollector/models/tasting.py` next to `wine.py`.
- The Pydantic schemas (`TastingCreate`, `TastingUpdate`, `TastingResponse`) live at `src/winecollector/schemas/tasting.py`.
- An Alembic migration creates the table with the FK index in V1.
- No router, service, or template references the model. A unit test asserts that `Wine.tastings` relationship loads as an empty list for a new wine.
