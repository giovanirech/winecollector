# ADR-003: Exportação JSON/CSV Manual no MVP

## Status

Accepted

## Date

2026-05-20

## Context

A tese central do WineCollector é "você é dono dos dados dos vinhos que comprou, imune à remoção do varejista". Existem dois caminhos de backup possíveis:

1. **Backup de infraestrutura**: rclone sincronizando `data/postgres/` e `data/images/` com Google Drive — já previsto em AGENTS.md.
2. **Exportação in-app**: botão que gera um arquivo portátil (JSON ou CSV) com todos os vinhos e suas fichas.

Sem o segundo caminho, o usuário depende de saber executar comandos rclone e entender que o dump do Postgres é o backup real — fricção alta para um produto que se posiciona como "seu arquivo pessoal".

## Decision

Incluir no MVP uma feature de **exportação manual JSON/CSV** acessível por um botão "Exportar Adega" no menu da aplicação. A exportação gera dois arquivos para download:

- **`adega.json`**: estrutura completa com todos os campos de cada vinho (incluindo URL original, status do scraping, estoque, datas).
- **`adega.csv`**: visão tabular dos campos principais (nome, vinícola, safra, uvas, região, estoque, data de compra) — para abrir em Excel/Google Sheets.

As imagens **não** são incluídas no arquivo de exportação — o JSON referencia o caminho relativo (`data/images/<uuid>.webp`) e o usuário usa o rclone para backupear o diretório de imagens em paralelo.

## Alternatives Considered

### Alternative 1: Confiar só no backup de infraestrutura (rclone)

- **Description**: MVP não tem exportação in-app. O usuário configura rclone para sincronizar `data/postgres/` e `data/images/` com Google Drive.
- **Pros**: simplicidade máxima — uma feature a menos para construir.
- **Cons**: enfraquece a tese central; usuário não-técnico não consegue inspecionar/migrar os próprios dados sem subir um Postgres; dump binário do Postgres não é "human-readable" e fica preso à versão do schema.
- **Why rejected**: contradiz o posicionamento de produto. O diferencial é justamente que o usuário tenha controle real e legível sobre os dados.

### Alternative 2: Deferir exportação para Fase 2

- **Description**: Sem exportação no MVP; marcar como prioridade alta da Fase 2.
- **Pros**: MVP entrega mais rápido.
- **Cons**: durante toda a vida do MVP a tese de preservação fica capenga; usuário começa a acumular vinhos sem ter como exportar — quando finalmente entrar na F2, já há volume e qualquer bug na exportação ganha gravidade.
- **Why rejected**: a exportação é parte do contrato do MVP. Adiar quebra a promessa para o usuário.

### Alternative 3: Incluir imagens no export (ZIP)

- **Description**: Exportação gera um ZIP contendo `adega.json`, `adega.csv` e a pasta `images/`.
- **Pros**: backup completo em um único arquivo.
- **Cons**: arquivos podem ficar grandes (centenas de MB) conforme a adega cresce; geração síncrona vira lenta; complica o fluxo de download.
- **Why rejected**: imagens já são backupeadas pelo rclone como diretório; manter o export leve evita travas na UI. Avaliar ZIP em fase futura se houver demanda.

## Consequences

### Positive

- Tese de "seus dados, sua propriedade" é validada e demonstrável no MVP.
- Arquivo JSON serve também como contrato de migração de schema entre versões.
- CSV permite que o usuário leve a adega para outras ferramentas (planilha, Notion, Airtable) sem depender do app.

### Negative

- Imagens ficam de fora do export — usuário precisa entender que rclone é o canal complementar para arquivos binários.
- Estrutura do JSON precisa ser estável o suficiente para ser útil em futuras importações (não temos importação na V1, mas o formato deve estar preparado).

### Risks

- **Risco**: formato JSON evolui entre versões e quebra exports antigos.
  - **Mitigação**: incluir um campo `schema_version` no topo do JSON desde a V1.
- **Risco**: exportação gera arquivo pesado se a adega crescer muito.
  - **Mitigação**: para o MVP single-user com adega típica (dezenas a poucas centenas de vinhos), o arquivo permanece pequeno (sub-MB). Re-avaliar se a escala mudar.

## Implementation Notes

- O JSON deve incluir um campo de metadata no topo: `{"schema_version": 1, "exported_at": "<iso8601>", "wines": [...]}`.
- A exportação é síncrona no MVP — gera o arquivo em memória e devolve como download. Para adegas grandes (futuras), considerar streaming.
- O CSV usa cabeçalho fixo em português para coincidir com a UI ("nome,vinícola,safra,uvas,região,estoque,data_compra").
