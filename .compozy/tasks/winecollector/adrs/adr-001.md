# ADR-001: Faseamento "Arquivista Primeiro"

## Status

Accepted

## Date

2026-05-20

## Context

O WineCollector tem três pilares de valor potencial: (1) **preservação** dos dados/imagens de vinhos antes que a wine.com.br os remova quando saem de linha, (2) **caderno de degustação** com notas sensoriais e memórias afetivas e (3) **gestão de estoque/drinking window** da adega.

Tentar entregar os três simultaneamente no MVP aumenta o risco de atrasar o pilar que o usuário identificou como central — preservação — e dilui o foco em uma aplicação cujo público é uma única pessoa. Por outro lado, um MVP "mínimo demais" (só catálogo + scraping) compromete a tese de "você é dono dos seus dados" porque deixa exportação e busca para fases distantes.

A escolha do faseamento define o ritmo de entrega, qual valor é validado primeiro e como o produto evolui após a V1.

## Decision

Adotar a abordagem **Arquivista Primeiro** com três fases:

- **MVP (Fase 1)**: cadastro de vinhos via URL com scraping resiliente do wine.com.br, fallback manual quando o scraping falha, ficha completa do vinho com imagem, controle simples de estoque (+1/-1), busca textual livre + filtros (uva/tipo, estoque) + ordenação (data de compra, safra), exportação JSON/CSV manual, autenticação single-user e UI mobile-first em português.
- **Fase 2**: degustações 1:N por vinho (notas sensoriais + memórias afetivas), histórico de degustações por vinho.
- **Fase 3**: página dedicada "Para Beber" com drinking window calculada a partir da safra + potencial de guarda, e estatísticas básicas da adega.

## Alternatives Considered

### Alternative 1: Caderno completo desde o dia 1

- **Description**: MVP entrega catálogo + scraping + degustações 1:N + página "Para Beber" + exportação simultaneamente.
- **Pros**: usuário recebe valor completo de uma vez; menor risco de "MVP esqueleto" que precisa muita iteração antes de ser útil.
- **Cons**: escopo grande aumenta o tempo até a primeira garrafa ser efetivamente preservada; degustação e drinking window competem por atenção antes do cadastro estar maduro; risco maior de retrabalho em features avançadas.
- **Why rejected**: o usuário identificou preservação como o pilar central. Empilhar features secundárias sobre um cadastro ainda não validado contradiz o YAGNI e atrasa exatamente o valor que motivou o projeto.

### Alternative 2: Inventário mínimo

- **Description**: MVP enxuto com apenas cadastro via URL + scraping + ficha + estoque. Sem busca avançada, sem exportação, sem degustação.
- **Pros**: time-to-first-feature minúsculo; cada fase posterior é incremental e pequena.
- **Cons**: adega com 20+ vinhos sem busca já é frustrante; exportação fora do MVP enfraquece a tese de "imune à remoção do varejista" — sem exportar, os dados ficam presos no Postgres local, dependentes só do backup de infra; produto pode parecer incompleto demais para uso real.
- **Why rejected**: a exportação JSON/CSV é parte fundamental da promessa de preservação ("você é dono dos dados"). Adiar isso compromete a tese central. Busca textual + filtros básicos também são essenciais para o uso típico (consultar antes de abrir uma garrafa).

## Consequences

### Positive

- MVP valida a tese central (preservação) antes de investir em features adjacentes.
- Exportação JSON/CSV no MVP reforça o diferencial de mercado ("imune à remoção do varejista").
- Escopo enxuto permite iteração rápida sobre o scraping (que é a parte mais frágil e novel do projeto).
- Cada fase tem critérios de sucesso claros antes de avançar para a próxima.

### Negative

- Usuário só pode registrar notas/memórias a partir da Fase 2 — para o usuário final que também valoriza o caderno de degustação, há uma espera.
- A página "Para Beber" só chega na Fase 3, embora o campo `potencial_de_guarda` já seja capturado pelo scraping desde a V1 (mas sem lógica de alerta).

### Risks

- **Risco**: usuário perde interesse em registrar degustações se a Fase 2 demorar muito.
  - **Mitigação**: a Fase 1 é deliberadamente enxuta para que a Fase 2 entre em sequência rápida (sem refatoração grande).
- **Risco**: campo `potencial_de_guarda` pode ser ruidoso/inconsistente no wine.com.br, dificultando a heurística da Fase 3.
  - **Mitigação**: durante a Fase 1 o scraper já tenta extrair o valor como inteiro (em anos), aplicando heurísticas simples ("5 anos" → 5, "10 a 15 anos" → 15, "Até 8 anos" → 8). Quando a extração é ambígua, o campo fica em branco e o usuário pode preencher manualmente. Isso evita uma migração de tipo (texto → inteiro) entre Fase 1 e Fase 3.

## Implementation Notes

- A modelagem de dados deve antecipar `WineTasting` como entidade 1:N desde a V1 (preparada no schema/migrations), mesmo que a UI só apareça na F2 — evita migrações dolorosas depois.
- A exportação JSON/CSV deve incluir todos os campos da ficha + caminho relativo da imagem; quando degustações forem adicionadas (F2), o formato é estendido sem quebrar exportações antigas.
