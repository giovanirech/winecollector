# ADR-008: Convert Wine Images to WebP via Pillow on Download

## Status

Accepted

## Date

2026-05-20

## Context

The scraper downloads bottle images from wine.com.br. The source format is typically JPEG, sometimes PNG. AGENTS.md specifies that images should be saved as `<uuid>.webp` in `data/images/`. We can either preserve the original format (and store `<uuid>.<ext>`) or normalize every image to WebP at download time.

WebP achieves ~25-35% smaller files than JPEG at equivalent visual quality. For a cellar of hundreds of wines, the absolute savings are modest (tens of MB), but the uniform format simplifies serving (one `<img>` tag template, one MIME type) and aligns with the AGENTS.md convention.

## Decision

Convert every downloaded image to **WebP** using Pillow on the server before saving to disk. The on-disk filename is always `<uuid>.webp`. The conversion runs in a thread-pool executor (`asyncio.to_thread`) to avoid blocking the event loop.

Conversion parameters:
- Quality: 85 (visually indistinguishable from source for product photos).
- Method: 4 (balanced encode speed vs. compression).
- Max dimensions: capped at 1600px on the longer edge (resize down if larger). Wine.com.br product images are typically already under this.

The original URL is preserved in the database (`Wine.image_url`) for traceability and re-scrape support.

## Alternatives Considered

### Alternative 1: Preserve original format

- **Description**: Save the image with its original extension (`<uuid>.jpg`, `<uuid>.png`, `<uuid>.webp` depending on source).
- **Pros**: no Pillow dependency; no CPU on conversion; zero risk of conversion artifacts.
- **Cons**: heterogeneous directory complicates templates (the `<img src>` needs the actual extension); larger storage footprint; contradicts AGENTS.md convention.
- **Why rejected**: the uniformity benefit outweighs the small Pillow cost.

### Alternative 2: Lazy conversion (serve original, convert on first access)

- **Description**: Save the original on download; convert to WebP on first request via a `/images/<uuid>` route.
- **Pros**: spreads CPU cost; possibly skips conversion for never-viewed images.
- **Cons**: adds a routing concern, caching layer, race conditions on first concurrent access; YAGNI for a single-user app.
- **Why rejected**: complexity not justified at this scale.

## Consequences

### Positive

- Every image on disk has a predictable extension — templates and exports use `<uuid>.webp` directly.
- Files are smaller; rclone syncs less data.
- Pillow is a common dependency that brings other capabilities (validation, dimension checks) for free.

### Negative

- One extra dependency (`pillow`).
- A small CPU cost on each new wine cadastro (~50ms on a modern machine).
- A handful of source images may have visible recompression artifacts at quality=85. The user can re-scrape with a different URL if it matters.

### Risks

- **Risk**: a maliciously crafted image triggers Pillow's known image-bomb path.
  - **Mitigation**: set `Pillow.Image.MAX_IMAGE_PIXELS` to a safe limit; the source is wine.com.br, not user uploads — attack surface is narrow.
- **Risk**: Pillow build dependencies cause issues on slim Docker images.
  - **Mitigation**: Pillow ships pre-built wheels for `python:3.12-slim`; no system packages needed.

## Implementation Notes

- The scraper service exposes `_save_image_as_webp(content: bytes) -> Path` that:
  1. Opens the bytes with `PIL.Image.open(BytesIO(content))`.
  2. Converts to RGB if needed (some PNGs are RGBA).
  3. Resizes if the longer edge > 1600px.
  4. Saves to `data/images/<uuid>.webp` with `format="WEBP", quality=85, method=4`.
  5. Returns the path.
- This function is invoked via `await asyncio.to_thread(...)` from the async scraper flow.
- The full `data/images/` directory is gitignored and lives outside the repo at runtime.
