# ADR-004: Cookie-Based JWT Transport for HTMX/Jinja2 UI

## Status

Accepted

## Date

2026-05-20

## Context

The WineCollector frontend is server-rendered Jinja2 with HTMX for interactions ‚ÄĒ there is no SPA, no separate JSON API client, and no need for cross-origin token sharing in V1. Authentication is provided by `fastapi-users`, which supports multiple JWT transports: `CookieTransport`, `BearerTransport`, or both combined behind a single authentication backend.

The transport choice determines how the browser carries credentials to each protected route, including HTMX-initiated requests. A wrong choice means writing JavaScript to inject `Authorization` headers on every HTMX call ‚ÄĒ exactly the kind of work the "no custom JavaScript" frontend rule forbids.

## Decision

Use **`CookieTransport`** with `fastapi-users`. The JWT is set as an HttpOnly, Secure (in production), SameSite=Lax cookie. Every HTMX or page request carries it automatically. There is no `Authorization` header involved in V1 and no public `/api` surface.

Cookie attributes:
- `httponly=True` ‚ÄĒ not readable by JavaScript.
- `secure=True` in production (`ENVIRONMENT=production`), `secure=False` in development.
- `samesite="lax"` ‚ÄĒ allows top-level navigation while blocking cross-site POSTs.
- `max_age` matches `ACCESS_TOKEN_EXPIRE_MINUTES`.

## Alternatives Considered

### Alternative 1: BearerTransport (header-based)

- **Description**: JWT carried in `Authorization: Bearer <token>` header on each request.
- **Pros**: Standard for API clients; trivially testable with `curl`.
- **Cons**: HTMX does not automatically inject headers ‚ÄĒ every HTMX element would need a custom JS hook (e.g., `htmx.config.headers`). Violates the "no custom JavaScript" rule. Storing the token in `localStorage` exposes it to XSS.
- **Why rejected**: friction with server-rendered HTMX flow; contradicts frontend rules.

### Alternative 2: Both transports combined

- **Description**: Cookie for the UI, Bearer for a future `/api/*`.
- **Pros**: Future-proof for external API consumers (e.g., a phone app or import script).
- **Cons**: Extra code and configuration with no concrete consumer in sight. YAGNI.
- **Why rejected**: no `/api` surface planned for V1 or Phase 2; can be added later as a second authentication backend without breaking the cookie one.

## Consequences

### Positive

- Zero JavaScript needed for authentication ‚ÄĒ every HTMX request just works.
- HttpOnly cookies block the most common XSS token-theft vector.
- Login form is a plain HTML form (`POST /auth/login`); logout is a simple `POST /auth/logout` that clears the cookie.

### Negative

- CSRF must be considered for state-changing requests. Mitigation: SameSite=Lax blocks cross-site POSTs by default; HTMX uses the same origin so it's unaffected.
- A future external API would need a second auth backend (cheap to add when it exists).

### Risks

- **Risk**: Cookie not sent on a request because of a SameSite misconfiguration.
  - **Mitigation**: integration test that authenticates and hits a protected route end-to-end.

## Implementation Notes

- Configure a single `AuthenticationBackend` with `CookieTransport` + `JWTStrategy`.
- `SECRET_KEY` (‚Č•32 chars) loaded from `.env` via `pydantic-settings`.
- Login route returns 204 + Set-Cookie; logout clears the cookie.
